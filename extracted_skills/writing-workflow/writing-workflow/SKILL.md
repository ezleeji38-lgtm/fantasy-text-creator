# 글쓰기 워크플로우 (Writing Workflow)

단편 콘텐츠를 위한 5단계 구조적 글쓰기 프로세스. 장편(책)은 book-writing-workflow, 에세이는 essay-workflow 스킬을 사용한다.

## 전체 흐름

```
Step 1: 목적 정의 → Step 2: 구조 설계 → Step 3: 초안 작성 → Step 4: 리라이팅 → Step 5: 최종 출력
```

사용자의 상태에 따라 적절한 단계부터 시작한다.
- 주제만 있음 → Step 1부터
- 구조/아웃라인 있음 → Step 3부터
- 초안 있음 → Step 4부터

### Step 간 전환 규칙

각 Step 완료 후, 다음 Step으로 넘어가기 전에 사용자 확인을 받는다.

| 전환 | 확인 방식 |
|------|----------|
| Step 1 → 2 | 목적·독자·채널·유형을 요약 제시하고, 확정 여부를 확인한 뒤 구조 설계로 진행한다 |
| Step 2 → 3 | 아웃라인을 제시하고, 구조 수정이 필요한지 확인한 뒤 초안 작성으로 진행한다 |
| Step 3 → 4 | 초안을 제시하고, 방향 수정이나 추가 요청이 있는지 확인한 뒤 리라이팅으로 진행한다 |
| Step 4 → 5 | 수정본을 제시하고, 추가 수정이 필요한지 확인한 뒤 최종 출력으로 진행한다 |

사용자가 "끝까지 진행해줘" 또는 "한 번에 해줘"라고 요청한 경우, 확인 없이 연속 진행한다.

## 공통 집필 원칙

모든 Step에서 아래 원칙을 적용한다. 각 Step의 세부 규칙은 이 원칙의 확장이며, 충돌 시 공통 원칙이 우선한다.

### 문체 기준
- 표준어 사용, 설명형·해설형 서술 중심
- 종결어미는 사용자 지정에 따름 (기본값: '~합니다'체)
- 문장은 18~25단어 내외 기본
- **한 문장에는 하나의 정보, 하나의 판단, 하나의 작업 설명만 담는다.** 이유·과정·결과를 한 문장에 모두 담지 않는다
- 문장은 능동형을 기본으로 하여 주체와 행위를 명확히 드러낸다
- 문단은 3~5문장을 기본으로 한다. 의미가 이어지는 문장은 반드시 하나의 문단으로 묶는다
- 줄바꿈은 문단 단위로만 적용. 문단 내부에서는 문장을 이어 작성. 줄바꿈은 다음 경우에만 허용한다: (1) 소제목 전환 (2) 작업 단계 전환 (3) 주제 블록이 명확히 달라질 때
- 감정적 표현, 감탄형 문장 지양
- 전체 톤은 차분하고 실무적인 문체 유지 (사용자 지정 톤이 있으면 우선)

### 서술 전략
- 글의 첫 문장: **"왜 이 주제가 중요한가"** 또는 **독자의 문제 상황**으로 시작. 기능·현상 나열로 시작하지 않음
- 설명 순서: **개념 → 판단 기준 → 실제 적용 방식**
- 소제목은 기능명이 아닌 **행동·판단 중심 문장**으로 작성
- 핵심 기준·판단 포인트는 **볼드체**로 강조
- "가능하다"보다 **"이렇게 판단한다"** 식의 단정적 서술
- 마무리는 핵심 요약 + 독자의 다음 행동(CTA) 제시

### 금지 요소
- 정보 나열식 구성 (리스트만 던지고 해설 없는 글)
- 결과물에 대한 감상형 표현
- 과도한 반복 설명
- 근거 없는 과장·미사여구

### 단어 선택 기준
- 쉽고 직관적인 단어를 우선 사용한다.
- 추상적 형용사보다 **행동·상태·결과를 설명하는 단어**를 사용한다. (❌ "효과적인 방법" → ✅ "텍스트 크기가 유지되는 방법")
- IT·AI·이미지 관련 전문 용어는 기능적 맥락 안에서만 사용한다.
- 과장 표현, 감정 형용사, 은유·비유 표현은 최소화한다.

### 분량 통제 규칙

수정·보강·추가 요청 시 원본 분량이 의도 없이 늘어나는 것을 방지하기 위해, 작업 동사별 허용 범위를 다음과 같이 구분한다. 이 규칙은 Step 3(초안 수정)과 Step 4(리라이팅) 모두에서 적용된다.

- **"다듬어줘" / "수정해줘"** = 원본의 문장 수·문단 수·전체 구조를 유지한 채 표현만 정리한다. 새로운 문장이나 문단을 추가하지 않는다.
- **"보강해줘"** = 지정된 부분에 한해 내용 추가를 허용한다. 미지정 부분은 원본을 유지한다.
- **"추가해줘"** = 지정된 위치에 지정된 분량만 삽입한다.
- 초안 작성 시 사용자가 분량을 지정하지 않으면, 콘텐츠 유형별 기본 분량을 기준으로 한다.
- 수정 결과물의 총 분량이 원본 대비 120%를 초과하면, 초과 사유를 사용자에게 먼저 확인한다.

### AI 초안 및 원고수정 시 적용 규칙
- AI 초안은 완성본이 아니라 **구조 재료**로 취급한다.
- 원고수정 단계에서는 문체 미화보다 **기준 명확화**를 우선한다.
- 기존 의미·논리는 유지하되 문단은 작업 단위로 재정렬한다.
- 문장은 읽히는 호흡 기준으로 정제한다.
- 최종 원고는 독자가 그대로 따라 할 수 있는 **작업 문서 수준**을 목표로 한다.

### AI 도구 활용 표기
- "AI가 대신 쓴다" 금지 → "AI를 활용해 설계한다"로 통일

## 스킬 간 연계

### 이 스킬의 범위

이 스킬은 **단편 콘텐츠**(1회 게시 단위의 글)를 처리한다. 다음은 이 스킬의 범위 밖이다.
- 장편(책) 집필 → book-writing-workflow
- 에세이(수필·산문) → essay-workflow
- 강의 슬라이드 제작 → lecture-slides-workflow

### 타 스킬 콘텐츠를 입력으로 받을 때

다른 스킬에서 생성된 콘텐츠를 이 스킬의 입력으로 사용하는 경우가 있다. 처리 방식은 다음과 같다.

| 입력 | 처리 방식 |
|------|----------|
| 책 챕터 → 블로그 글로 축약 | 원본 챕터의 핵심 논지를 추출하고, 이 스킬의 유형별 구조 템플릿에 맞춰 재구성한다. 원본의 문체가 아닌 이 스킬의 채널별 톤을 적용한다 |
| 책 챕터 → SNS 롱폼으로 변환 | 핵심 판단 기준 1~2개만 뽑아 경험 중심 서술로 전환한다. 원본 분량의 30~50%를 목표로 한다 |
| 강의 스크립트 → 블로그 글로 변환 | 구어체를 문어체로 전환하고, 시간 순서 구성을 논리 순서 구성으로 재배치한다 |
| 에세이 → 칼럼으로 변환 | 개인 경험 중심 서술에서 논증 구조로 전환한다. 관찰→주장→근거→결론 순서를 적용한다 |

원본 콘텐츠의 핵심 내용은 유지하되, 구조·톤·분량은 이 스킬의 대상 유형 기준을 따른다.

### 이 스킬의 콘텐츠를 타 스킬로 보낼 때

이 스킬에서 작성한 글을 다른 스킬의 입력으로 사용하는 경우, 이 스킬의 역할은 최종 출력(Step 5)까지 완료하는 것이다. 이후 변환 작업은 대상 스킬이 담당한다.

## Step 1: 목적 정의 (Purpose Definition)

글을 쓰기 전에 다음 4가지를 명확히 한다.

### 수집 항목

| 항목 | 질문 | 예시 |
|------|------|------|
| **목적** | 이 글로 무엇을 달성하려는가? | 설득, 교육, 정보 공유, 브랜딩, 기록 |
| **독자** | 누가 읽는가? 현재 수준은? | 입문자, 실무자, 의사결정자, 일반 대중 |
| **채널** | 어디에 게시하는가? | 블로그, 뉴스레터, 링크드인, 사내 보고 |
| **콘텐츠 유형** | 어떤 형식인가? | 아래 유형 중 선택 |

### 콘텐츠 유형 분류

| 유형 | 특징 | 기본 분량 |
|------|------|----------|
| 블로그/아티클 | SEO 고려, 스캔 가능한 구조 | 1,500~3,000자 |
| 뉴스레터 | 개인 톤, 정기 발행, CTA 포함 | 800~1,500자 |
| 칼럼/기고문 | 관점 중심, 논증 구조 | 1,500~2,500자 |
| 분석 기사/심층 아티클 | 주제 심층 분석, 사례 배치, 판단 기준 제시 | 4,000~6,000자 |
| 보고서/제안서 | 구조적, 데이터 기반, 의사결정 지원 | 2,000~5,000자 |
| 보도자료 | 역피라미드, 객관적 톤 | 800~1,200자 |
| SNS 롱폼 | 브런치/링크드인, 경험 중심 | 1,000~2,000자 |
| 강의 스크립트 | 구어체, 시간 단위 구성 | 분당 약 300자 |

유형이 결정되면 Step 2에서 해당 유형의 구조 템플릿을 적용한다.
상세 템플릿은 `references/content-templates.md`를 참고한다.

### 산출물

4가지 항목을 요약 정리해 사용자에게 제시한다. 사용자 확인 후 Step 2로 진행한다.

## Step 2: 구조 설계 (Structure Design)

Step 1에서 확정된 유형별 구조 템플릿을 기반으로 아웃라인을 작성한다.

### 유형별 구조 골격

**블로그/아티클:**
```
도입: 독자의 문제 또는 질문 제기
본론 1: 핵심 개념과 판단 기준
본론 2: 실제 적용 방식 + 예시
본론 3: 자주 발생하는 실수 또는 심화
마무리: 핵심 정리 + CTA
```

**뉴스레터:**
```
인사/도입: 이번 호의 핵심 메시지
메인 콘텐츠: 1~2개 주제 (간결하게)
부가 정보: 추천 자료, 링크, 팁
마무리: CTA + 다음 호 예고
```

**칼럼/기고문:**
```
도입: 논점 제기 (사건, 현상, 질문)
논증 1: 근거 + 해석
논증 2: 반론 검토 또는 보강
결론: 저자의 판단 + 제언
```

**분석 기사/심층 아티클:**
```
도입: 현상 또는 문제의 재정의
본론 1: 핵심 개념과 판단 기준 (왜 다른가, 왜 중요한가)
본론 2: 구조/인프라 설명 (기준을 작동시키는 메커니즘)
본론 3: 실전 사례 (기준이 작업에서 어떻게 작동하는가)
마무리: 관점 재정렬 (요약이 아닌, 질문의 전환)
```

**보고서/제안서:**
```
요약(Executive Summary)
배경/현황
분석/핵심 내용
제안/권고사항
기대 효과
부록 (데이터, 참고 자료)
```

나머지 유형 상세 구조는 `references/content-templates.md`를 참고한다.

### 산출물

아웃라인을 사용자에게 제시한다. 복잡한 글의 경우 별도 문서로 생성할 수 있다: `outline.docx`. 사용자 확인 후 Step 3으로 진행한다.

## Step 3: 초안 작성 (First Draft)

### 초안 작성 규칙

- Step 2에서 확정된 구조를 따라 작성한다
- 공통 집필 원칙(서술 전략, 문체 기준)을 적용한다
- 문단은 4~6줄 이내 유지
- 완성도보다 흐름 우선 — 막히는 부분은 `[TODO: ...]` 마커 삽입
- 채널에 맞는 톤 적용 (아래 톤 가이드 참고)

### 톤 가이드

| 채널/유형 | 톤 | 특징 |
|----------|-----|------|
| 블로그 | 친근한 전문가 | 쉬운 설명, 경험 공유, 독자에게 말 거는 느낌 |
| 뉴스레터 | 신뢰하는 동료 | 개인적 어조, 솔직함, 큐레이션 |
| 칼럼 | 분석적 관찰자 | 근거 기반, 절제된 표현, 명확한 관점 |
| 분석 기사 | 구조적 해석자 | 현상 재정의, 판단 기준 제시, 깊이 있는 사례 배치 |
| 보고서 | 객관적 전문가 | 데이터 중심, 구조적, 판단 근거 명시 |
| 보도자료 | 공식 발표 | 3인칭, 객관적, 사실 중심 |
| SNS 롱폼 | 경험을 나누는 실무자 | 스토리텔링, 교훈 도출, 공감 유도 |

### 산출물

- 초안(draft)은 **대화창 텍스트로 직접 제공**한다 (파일 생성하지 않음)
- 사용자가 파일을 명시적으로 요청한 경우에만 `draft.docx` 또는 `draft.md`로 저장한다
- 사용자 확인 후 Step 4로 진행한다

## Step 4: 리라이팅 (Rewriting)

초안을 4가지 관점에서 점검하고 수정한다. 점검 → 리포트 → 확인 → 수정의 순서로 진행한다.

### 진행 방식

1. **점검 리포트 제시** — 4가지 점검 결과를 표 형태로 사용자에게 보여준다
2. **사용자 확인** — 수정 방향에 대해 사용자의 동의 또는 추가 지시를 받는다
3. **수정본 작성** — 확인된 방향에 따라 전체 수정본을 제시한다
4. **변경 요약** — 수정본 하단에 주요 변경 사항을 간결하게 정리한다

사용자가 "리라이팅해줘"라고만 요청한 경우, 1~4를 한 번에 진행한다. "점검만 해줘"라고 요청한 경우, 1까지만 진행하고 대기한다.

### 점검 리포트 표준 포맷

각 점검 항목을 아래 형식으로 출력한다.

| # | 점검 항목 | 판정 | 수정 방향 |
|---|----------|------|----------|
| 1 | (항목) | O / 수정 필요 | (수정 필요 시 구체적 방향 기술) |

### 점검 1: 구조 점검

| # | 점검 항목 | 수정 방향 (문제 발견 시) |
|---|----------|------------------------|
| 1 | 도입부가 독자의 문제/질문으로 시작하는가 | 기능·현상 나열로 시작했다면, 독자가 겪는 구체적 상황이나 질문으로 교체한다. 예: ❌ "클로드는 장문맥을 지원한다" → ✅ "10개 챕터를 쓰는데 3장의 맥락이 7장에서 사라진다면" |
| 2 | 전체 흐름이 개념 → 기준 → 적용 순서를 따르는가 | 적용(사례)이 기준(판단 근거) 앞에 나왔다면 순서를 재배치한다. 기준 없이 사례만 나열된 섹션은 판단 기준 문단을 추가한다 |
| 3 | 소제목이 행동·판단 중심 문장인가 | 기능명·명사 나열형 소제목을 판단 중심 문장으로 전환한다. 예: ❌ "클로드의 5가지 강점" → ✅ "왜 다른 도구와 다른가" |
| 4 | 불필요한 섹션이나 중복 내용이 없는가 | 같은 내용이 다른 표현으로 반복되면 한 곳에 통합한다. 두 섹션이 같은 강점을 설명하면 하나를 삭제하고 나머지에 깊이를 더한다 |
| 5 | 마무리에 핵심 요약과 CTA가 있는가 | 요약 없이 끝나면 핵심 판단 기준 1~2문장을 추가한다. CTA는 채널 특성에 맞춰 설정한다 (블로그: 행동 유도, 칼럼: 관점 재정렬) |

### 점검 2: 문장 점검

| # | 점검 항목 | 수정 방향 (문제 발견 시) |
|---|----------|------------------------|
| 1 | 문장이 18~25단어 범위 내인가 | 25단어 초과 문장은 두 문장으로 분리한다. 이유·과정·결과가 한 문장에 있으면 각각 독립 문장으로 나눈다 |
| 2 | 종결어미가 통일되어 있는가 | ~다체/~합니다체 혼용이 발견되면 글 전체의 기준 어미로 통일한다 |
| 3 | 단정적 서술을 사용하는가 | "~할 수 있다", "~가능하다"를 "~한다", "~된다"로 전환한다. 예: ❌ "맥락을 유지할 수 있다" → ✅ "맥락을 유지한다" |
| 4 | 감정적/감탄형/과장 표현이 없는가 | 감탄형("놀라운", "획기적인"), 과장("완벽하게"), 감정 호소("꼭 해보세요")를 삭제하고 상태·결과 서술로 대체한다 |
| 5 | 핵심 포인트가 볼드 강조되어 있는가 | 각 섹션에서 판단 기준이 되는 문장 1~2개를 볼드 처리한다. 3개 이상 볼드는 강조 효과가 희석되므로 줄인다 |
| 6 | 문단 간 전환이 자연스러운가 | 번호 나열("첫째", "둘째")로만 연결된 문단은 앞 문단의 결론이 다음 문단의 전제가 되도록 연결 문장을 추가한다 |

### 점검 3: 스타일 점검

홍작가 글쓰기 스타일 기준과의 일치 여부를 점검한다.

| # | 점검 항목 | 수정 방향 (문제 발견 시) |
|---|----------|------------------------|
| 1 | 방법·팁·요령이 기준 설명 없이 먼저 등장하지 않는가 | 방법이 먼저 나왔다면, 앞에 "왜 이 방식인가"를 설명하는 기준 문단을 추가한다 |
| 2 | 감정적 공감이나 동기 부여 문장이 논리 설명보다 앞서지 않는가 | 공감형 도입("많이 힘드시죠?")을 상황 진단형("구조가 정리되지 않은 상태다")으로 교체한다 |
| 3 | 글을 읽은 뒤 판단 기준의 변화가 느껴지는가 | 정보 전달로만 끝난 글이면, 마무리에 관점 재정렬 문장을 추가한다. "~하는 법"이 아니라 "~로 질문을 바꾸는 것"으로 끝맺는다 |
| 4 | 어휘가 기능·도구 중심이 아닌 기준·판단·구조 중심인가 | "기능", "도구", "방법"이 과다하면 "기준", "판단", "구조", "설계"로 대체 가능한 부분을 교체한다 |
| 5 | 문체 리듬이 점층 구조를 따르는가 | 문단 초반부터 판단·결론이 나오면, 앞에 관찰·상황 제시 문장을 배치해 뒤로 갈수록 밀도가 높아지도록 조정한다 |

### 점검 4: 채널 최적화

채널 유형에 따라 해당 항목만 점검한다.

| # | 점검 항목 | 적용 채널 | 수정 방향 (문제 발견 시) |
|---|----------|----------|------------------------|
| 1 | 분량이 채널 기준에 맞는가 | 전체 | Step 1에서 정한 유형별 기본 분량과 비교한다. ±20% 이내로 조정한다 |
| 2 | 톤이 채널에 적합한가 | 전체 | Step 3 톤 가이드와 대조해 이탈한 문단을 수정한다 |
| 3 | 제목/소제목이 매력적이고 명확한가 | 전체 | 제목 후보 3~5개를 제시하고, 소제목은 스캔 가능성을 기준으로 점검한다 |
| 4 | SEO 키워드가 자연스럽게 반영되었는가 | 블로그/아티클 | 제목, 도입부, 소제목에 핵심 키워드가 포함되었는지 확인한다 |
| 5 | CTA가 목적에 부합하는가 | 블로그, 뉴스레터, SNS | 채널별 CTA 형태를 확인한다 (블로그: 행동 유도, 뉴스레터: 구독/공유, 칼럼: 관점 제시로 마무리) |
| 6 | 인용·출처가 명시되어 있는가 | 보고서, 칼럼, 보도자료 | 데이터·사실관계에 출처가 빠져 있으면 추가하거나 [TODO] 마커를 삽입한다 |

## Step 5: 최종 출력 (Final Output)

Step 4 수정본을 출판/게시 수준으로 마무리한다.

### 진행 방식

1. **최종 점검 실행** — 아래 항목을 점검하고 결과를 사용자에게 제시한다
2. **제목 확정** — 후보를 제시하고 사용자가 선택한다
3. **사용자 확인** — 최종본 텍스트를 보여주고 확정 여부를 확인한다
4. **파일 생성** — 확정 후 지정된 포맷으로 파일을 생성한다

사용자가 "최종 출력해줘"라고만 요청한 경우, 1~4를 한 번에 진행한다.

### 최종 점검 항목

| # | 점검 항목 | 기준 |
|---|----------|------|
| 1 | 제목 확정 | 후보 3~5개를 생성한다. 기준: (1) 핵심 키워드 포함 (2) 15자 이내 권장 (3) 글의 톤과 일치 (4) 독자의 문제 또는 판단 기준이 드러남. 사용자가 선택한다 |
| 2 | 소제목 정리 | 스캔 가능하고 행동·판단 중심인지 최종 확인한다 |
| 3 | 맞춤법·문법 교정 | 한국어 맞춤법, 띄어쓰기, 외래어 표기법, 숫자 표기(만 단위 쉼표 등), 문장부호 사용을 점검한다 |
| 4 | 메타 정보 (블로그) | 메타 디스크립션(120자 이내), 태그(5~10개), 카테고리를 작성한다. 블로그 외 채널은 생략 |
| 5 | [TODO] 마커 해결 | 모든 `[TODO]` 마커가 해결 또는 제거되었는지 확인한다 |
| 6 | 분량 최종 확인 | Step 1에서 정한 유형별 기본 분량 범위(±20%) 안에 있는지 확인한다 |
